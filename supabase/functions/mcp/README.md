# ChainWork MCP server (`functions/mcp`)

A [Model Context Protocol](https://modelcontextprotocol.io) server, hosted **free**
as a Supabase Edge Function, that lets **Claude Code** read a chain's tasks, work
on them, and mark them done.

> The actual coding + `git push` is Claude Code's own job. This server is only the
> ChainWork bridge: it exposes your chains/tasks as MCP tools.

## Tools exposed

| Tool | What it does |
| --- | --- |
| `whoami` | Confirm the connection / see which user the key belongs to |
| `list_chains` | List your chains (id, name, 8-char code, role) |
| `list_tasks` | List tasks of a chain (filter by status / project / chain-level) |
| `get_next_task` | The "next" task — `next_pending` (default) or `newest` |
| `get_task` | Full detail of one task (incl. project repo links) |
| `create_task` | Create a task |
| `update_task` | Edit title / description / priority / due date / assignees / status |
| `set_task_status` | `pending` / `in_progress` / `done` |
| `complete_task` | Mark done (+ optional note as a comment) and notify the chain |
| `add_comment` | Comment on a task |

## Auth

Each request must carry a **ChainWork API key** (`cw_live_…`), generated by the
user in **Settings → Integrations**. Present it as either:

- `Authorization: Bearer cw_live_…`  (recommended), or
- `X-ChainWork-Key: cw_live_…`

The function SHA-256-hashes the key, looks it up in `public.mcp_tokens` (table
created by migration `0027_mcp_tokens.sql`) using the service role, and then acts
scoped to that user, re-checking `chain_members` membership on every call.

## Deploy (no CLI needed)

This repo runs migrations by pasting them into the Supabase **SQL editor**, and
you can deploy this function the same hands-off way from the dashboard:

1. Apply migration `supabase/migrations/0027_mcp_tokens.sql` in the SQL editor.
2. Supabase Dashboard → **Edge Functions** → **Deploy a new function** → name it
   **`mcp`** → paste the contents of `index.ts`.
3. **Turn OFF "Verify JWT"** for this function (toggle in the function's settings,
   or `verify_jwt = false`). This is required so the gateway forwards our own
   `Authorization` header instead of demanding a Supabase JWT.
4. No extra secrets to set — `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` are
   injected automatically.

> Prefer the CLI? `supabase functions deploy mcp --no-verify-jwt`.

The endpoint is then:

```
https://<project-ref>.supabase.co/functions/v1/mcp
```

## Connect from Claude Code

Generate a key in **Settings → Integrations** (it's shown once), then run:

```bash
claude mcp add --transport http chainwork \
  https://<project-ref>.supabase.co/functions/v1/mcp \
  --header "Authorization: Bearer cw_live_xxxxxxxx"
```

Now just talk to Claude Code:

> "De la chain `AB12CD34`, hacé la próxima tarea pendiente, pusheá los cambios a
> GitHub y marcala como completada."

Claude Code will call `get_next_task` → do the work → `git push` (its own git) →
`complete_task`. Any open ChainWork tab updates live, and the chain gets the
"task completed" notification.

## Troubleshooting

- **401 even with a fresh key** — make sure migration `0027` is applied and the key
  was generated in *this* project. Keys are `cw_live_…` and SHA-256-matched.
- **Gateway rejects the request / "missing authorization"** — confirm **Verify JWT
  is OFF** for the `mcp` function. If your gateway still inspects `Authorization`,
  pass the key in a custom header instead:
  `--header "X-ChainWork-Key: cw_live_xxx"` (the function accepts either).
- **Claude Code says it can't connect** — verify the URL ends in
  `/functions/v1/mcp` and test it with the curl smoke test below.

## Smoke test

```bash
curl -s -X POST https://<project-ref>.supabase.co/functions/v1/mcp \
  -H "Authorization: Bearer cw_live_xxxxxxxx" \
  -H "Content-Type: application/json" \
  -d '{"jsonrpc":"2.0","id":1,"method":"tools/call","params":{"name":"whoami","arguments":{}}}'
```
